# Resonance-update-chk 작업 규칙

이 파일은 `C:\CODEX\Resonance-update-chk` 전용 작업 규칙이다. 레조넌스 업데이트 여부 확인, 변경점 점검, 업데이트 관련 원본/리포트 정리에만 적용한다.

## 기본 원칙

- 사용자와의 대화, 작업 설명, 최종 보고는 한국어 존댓말로 작성하고 반말하지 않는다.
- 작업 시작 전 이 폴더의 기존 파일, README, 스크립트, 산출물을 먼저 확인한다.
- 변경 범위는 업데이트 확인 작업에 필요한 파일로 제한한다.
- 사용자가 만들거나 수정한 기존 파일을 임의로 되돌리지 않는다.
- 한국어/중국어/일본어가 포함된 파일은 UTF-8로 읽고 쓴다.

## 프로젝트 범위

- 이 폴더는 레조넌스 공식/관련 소스의 업데이트 상태를 확인하고 변경점을 기록하는 별도 작업공간이다.
- `C:\CODEX\Resonance-Updater`, `C:\CODEX\resonance-db-review`, `C:\CODEX\resonance-story`는 기본적으로 참조 전용이다.
- 기존 프로젝트 파일을 수정해야 할 경우, 사용자의 명시 요청을 먼저 확인한다.
- 업데이트 확인 결과는 원본 데이터와 사람이 읽는 리포트를 분리해서 보관한다.

## 파일 배치

- 원본 수집물은 가능하면 `data/raw/YYYY-MM-DD/` 또는 `snapshots/YYYY-MM-DD/` 아래에 둔다.
- 정리된 비교 결과와 사람이 읽을 보고서는 `reports/` 아래에 둔다.
- 반복 실행할 수 있는 수집/비교 로직은 `tools/` 아래 Node 스크립트로 둔다.
- 수집 데이터에는 가능한 범위에서 출처 URL, 수집 시각, 요청 기준, 비교 기준 버전을 함께 남긴다.
- 임시 확인 파일은 최종 산출물과 섞지 않고 필요 없어지면 정리한다.

## 소스 확인 규칙

- 공식 사이트, 공식 API, 배포 파일, 앱/패키지 메타데이터처럼 1차 출처를 우선한다.
- 커뮤니티 글이나 보조 문서를 사용할 때는 보조 출처임을 리포트에 명시한다.
- 날짜와 버전은 상대 표현 대신 가능한 한 `YYYY-MM-DD`와 원문 버전 문자열로 기록한다.
- 자동화 스크립트가 네트워크를 사용하면 실패 원인과 재시도 조건을 리포트에 남긴다.

## Git 규칙

- 이 폴더가 Git 저장소로 초기화되어 있으면 작업 전후 `git status --short`를 확인한다.
- 기본적으로 로컬 변경만 수행한다.
- `push`는 사용자가 명시적으로 요청한 경우에만 수행한다.
- `main` 브랜치에 직접 커밋하지 않는다.

## 확인 규칙

- 스크립트나 데이터 파이프라인을 추가/수정하면 가능한 범위에서 직접 실행해 확인한다.
- 실행이 네트워크, 권한, 외부 서비스 문제로 막히면 정확한 명령과 에러를 기록한다.
- 단순 문서 설정만 바꾼 경우에는 파일 존재와 내용 확인으로 검증한다.
